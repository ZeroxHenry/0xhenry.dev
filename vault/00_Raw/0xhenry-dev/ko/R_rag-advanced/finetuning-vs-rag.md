### 서론

AI 엔지니어링에서 가장 많이 받는 질문 중 하나는 바로 이것입니다: **"내 데이터를 학습시키기 위해 파인튜닝을 해야 할까요, 아니면 RAG를 써야 할까요?"**

GPT-3 초기 시절에는 모두가 파인튜닝이 정답이라고 생각했습니다. 하지만 오늘날의 합의는 달라졌습니다. 대부분의 개발자에게는 RAG가 필요하지만, 특정 유즈케이스에서는 여전히 파인튜닝이 강력한 힘을 발휘합니다. 이 둘의 차이를 명확히 아는 것이 프로젝트의 예산과 성능을 결정짓는 핵심입니다.

---

### RAG (Retrieval-Augmented Generation): 오픈북 테스트

RAG는 AI에게 **교과서**와 검색 엔진을 쥐여주는 것과 같습니다. 모델은 데이터를 자신의 '가중치(Weights)'에 저장하지 않고, 필요할 때마다 실시간으로 찾아봅니다.

-   **적합한 작업**: 자주 변하는 지식 (예: 뉴스, 고객 서비스 문서, 실시간 재고량).
-   **장점**:
    -   **동적**: 벡터 DB에 파일 하나만 바꿔 넣으면 지식이 즉시 업데이트됩니다.
    -   **인용 가능**: AI가 답변을 생성할 때 어떤 문서를 참고했는지 정확히 말해줄 수 있습니다.
    -   **저비용**: 초기 설정 이후에는 고가의 GPU나 복잡한 학습 과정이 필요하지 않습니다.
-   **단점**: 검색 오류 리스크. 검색 결과가 부실하면 답변도 부실해집니다.

### 파인튜닝 (Fine-tuning): 전문 기술 습득

파인튜닝은 **외과 의사를 수련**시키는 과정과 비슷합니다. 모델의 내부 신경망 가중치를 직접 수정하여 특정 스타일, 포맷, 혹은 아주 좁은 도메인의 전문성을 학습시킵니다.

-   **적합한 작업**: 특정 스타일 학습(예: "0xHenry처럼 글쓰기"), 특정 출력 포맷 강제(예: "항상 SQL 쿼리로만 답변하기"), 전문 용어가 난무하는 특수한 도메인(예: 의료 진단).
-   **장점**:
    -   **패턴 매칭**: 특정 작업에 대해 비약적으로 높은 성능을 보입니다.
    -   **복잡성 처리**: 프롬프트만으로는 설명하기 힘든 미묘한 차이를 학습할 수 있습니다.
-   **단점**: 정적인 지식. 한 번 학습된 모델은 학습 시점 이후의 새로운 데이터를 알지 못합니다. 또한 학습 과정은 비용이 많이 들고 속도가 느립니다.

---

### 의사결정 매트릭스

| 특징 | RAG 사용 시점 | 파인튜닝 사용 시점 |
| :--- | :--- | :--- |
| **새로운 지식 업데이트** | 매우 유리 (실시간) | 불리 (정적) |
| **톤앤매너/스타일 구현** | 보통 | 매우 유리 |
| **할루시네이션 위험** | 낮음 (근거 명시) | 높음 (학습 데이터 누락 시) |
| **구축 비용** | 낮음~보통 | 높음 |
| **응답 속도(Latency)** | 보통 (검색 단계 포함) | 빠름 (추론만 수행) |

### 하이브리드 방식: 왜 둘 다 쓰지 않나요?

2026년 현재, 가장 뛰어난 상용 시스템은 **두 방식을 모두** 사용합니다.
1.  **파인튜닝**: 소형 모델(예: Gemma 2 9B)이 업계 전문 용어와 출력 포맷을 완벽히 이해하도록 학습시킵니다.
2.  **RAG**: 학습된 소형 모델 위에 RAG 시스템을 얹어, 항상 최신의 정확한 팩트를 제공하도록 설계합니다.

### 요약

RAG는 **지식(Facts)**을 위한 것이고, 파인튜닝은 **양식(Form)**을 위한 것입니다. AI가 5분 전의 정보를 알기를 원한다면 RAG를 쓰세요. AI가 특정 인물처럼 행동하거나 아주 엄격한 기술적 구조를 따르기를 원한다면 파인튜닝을 쓰세요.
