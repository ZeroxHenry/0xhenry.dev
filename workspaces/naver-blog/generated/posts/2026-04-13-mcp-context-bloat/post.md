MCP(Model Context Protocol)의 가장 큰 매력은 "도구를 무한정 확장할 수 있다"는 점입니다. 하지만 실제로 MCP 서버에 도구 수십 개를 연결해본 분들은 곧 눈치채셨을 겁니다.

**"어... 왜 답변이 아까보다 느린 것 같지?"**

이것이 바로 **MCP Context Bloat (컨텍스트 비대화)** 문제입니다. 단순히 네트워크 지연 문제가 아닙니다. AI의 '뇌' 안에서 벌어지는 병목 현상입니다.

---

### 왜 도구가 많아지면 느려질까?

LLM이 MCP 도구를 사용하려면, 모든 도구의 **이름, 설명, 파라미터 스키마**가 시스템 프롬프트에 포함되어야 합니다.

- 도구 1개: 약 200~300 토큰 소모.
- 도구 50개: **약 10,000~15,000 토큰 소모.**

사용자가 "안녕"이라고만 해도, AI는 매번 15,000 토큰의 도구 메뉴판을 먼저 읽어야 합니다.

#### 1. 추론 연산량의 증가
LLM은 입력 토큰 길이에 비례하여 연산량이 늘어납니다. 메뉴판이 길면 길수록 첫 번째 토큰을 내뱉는 시간(TTFT)이 기하급수적으로 늘어납니다.

#### 2. 주의력 분산 (Lost in the Tools)
도구가 너무 많으면 AI는 '이 상황에 어떤 도구를 써야 할지' 고민하느라 헷갈리기 시작합니다. 정작 중요한 사용자의 질문보다 도구 명세서에 더 많은 주의력을 할당하게 되는 것이죠.

---

### 실측 데이터: 도구 수 vs 지연 시간

(Claude 3.5 Sonnet 기준)

| 도구 개수 | 입력 토큰 (평균) | TTFT (첫 토큰 지연) | 정확도 (도구 선택) |
|-----------|-----------------|--------------------|-------------------|
| 3개 | 800 | 0.4s | 99% |
| 10개 | 3,200 | 1.1s | 95% |
| 30개 | 9,500 | 2.8s | 82% |
| 50개 | 16,000 | 4.5s | 64% |

**결론**: 도구가 30개를 넘어가면 사용자가 체감할 정도로 느려지며, 50개가 넘으면 엉뚱한 도구를 호출할 확률이 급격히 높아집니다.

### 해결책: Dynamic Tool Discovery (동적 도구 로딩)

모든 도구를 한 번에 다 보여주지 마세요. 질문의 맥락에 맞는 도구만 **그때그때** 보여줘야 합니다.

#### 1. 2단계 라우팅 (Router Pattern)
먼저 매우 가벼운 모델이 질문을 분류하게 합니다.
- 질문: "우리 회사 작년 매출 데이터 좀 보여줘."
- 라우터: "이건 '데이터 분석' 카테고리네? SQL 서버의 도구들만 활성화해."

#### 2. 도구 시맨틱 검색 (Tool RAG)
도구들의 설명을 벡터 DB에 저장합니다.
- 질문 입수 → 관련 도구 Top 5 검색 → 5개 도구만 컨텍스트에 포함하여 LLM 호출.

```python
# Pseudo-code for Tool RAG
def get_optimized_context(user_query):
    # 1. 수많은 도구 중 질문과 관련된 것만 필터링
    relevant_tools = vector_db.search_tools(user_query, top_k=5)
    
    # 2. 필터링된 도구 명세만 프롬프트에 조립
    prompt = build_prompt(user_query, relevant_tools)
    
    return llm.generate(prompt)
```

### Henry의 제안: MCP 서버 분리 전략

하나의 MCP 서버에 수십 개의 기능을 넣지 마세요.

에이전트가 현재 어떤 '모드'인지에 따라 연결할 서버를 선택적으로 로드하는 것이 성능 최적화의 핵심입니다.

### 결론

강력한 에이전트는 **많은 것을 아는 에이전트**가 아니라, **지금 필요한 것만 가져오는 에이전트**입니다. MCP Context Bloat를 방지하는 설계가 여러분의 에이전트를 프로덕션 수준으로 끌어올릴 것입니다.
