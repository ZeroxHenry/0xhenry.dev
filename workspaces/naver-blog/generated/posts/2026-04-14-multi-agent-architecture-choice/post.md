에이전트 하나로 부족할 때, 우리는 '에이전트 팀'을 만듭니다. 이때 가장 먼저 마주하는 질문은 **"이 팀을 누가 관리하게 할 것인가?"**입니다. 

오늘은 멀티 에이전트 아키텍처의 양대 산맥인 **Supervisor 패턴**과 **Swarm 패턴**의 장단점을 파헤쳐 봅니다.

---

### 1. Supervisor 패턴: 중앙 집권형 아키텍처

팀장(Supervisor) 에이전트가 모든 요청을 받고, 이를 하위 에이전트들에게 분배한 뒤 결과를 취합하는 방식입니다.

- **작동 방식**: 사용자 → Supervisor → (Agent A / Agent B / Agent C) → Supervisor → 사용자
- **장점**: 
    - **통제력**: 전체 워크플로우를 관리하기 쉽고, 결과물의 품질이 일정합니다.
    - **보안**: 하위 에이전트들이 서로의 존재를 알 필요가 없어 권한 관리가 용이합니다.
- **단점**: 
    - **병목 현상**: Supervisor 에이전트에게 너무 많은 추론 부담이 실립니다.
    - **비용**: 모든 단계에서 Supervisor를 거쳐야 하므로 토큰 소모가 큽니다.

---

### 2. Swarm 패턴: 자율 분산형 아키텍처 (Handoff)

OpenAI의 'Swarm' 프레임워크로 유명해진 방식입니다. 에이전트들이 대등한 관계에서 필요에 따라 작업을 서로 "넘겨주는(Handoff)" 방식입니다.

- **작동 방식**: 사용자 → Agent A → (작업 위임) → Agent B → 사용자
- **장점**: 
    - **확장성**: 새로운 에이전트를 추가하기 쉽고 구조가 유연합니다.
    - **속도**: 중앙 관리자를 거치지 않고 직접 넘겨주므로 반응이 빠릅니다.
- **단점**: 
    - **예측 가능성**: 에이전트들끼리 작업을 핑퐁(Ping-pong)하며 무한 루프에 빠질 위험이 큽니다.
    - **디버깅**: 작업이 어디로 튀었는지 추적하기가 상대적으로 어렵습니다.

---

### 무엇을 선택해야 할까? (결정 트리)

1. **워크플로우가 정형화되어 있는가?**
   - YES → **Supervisor 패턴** (계획된 루틴을 수행하기에 적합)
   - NO → **Swarm 패턴** (탐색적이고 자율적인 작업에 적합)

2. **품질 검수가 결정적인가?**
   - YES → **Supervisor 패턴** (관리자가 최종 답변을 검수할 수 있음)
   - NO → **Swarm 패턴** (에이전트의 개별 능력을 믿고 진행)

---

### Henry의 구현 팁: "하이브리드가 정답이다"

프로덕션 환경에서는 보통 이 둘을 섞어 씁니다. 전체 큰 흐름은 **Supervisor**가 잡고, 특정 전문적인 서브 태스크(예: 복잡한 코드 리뷰)만 **Swarm** 방식으로 에이전트들끼리 토론하게 만드는 것이 효율과 안정성을 모두 챙기는 방법입니다.

---

### 결론

아키텍처는 정답이 아니라 **'타협(Trade-off)'**의 결과입니다. 우리 팀의 작업이 '군대식 명령 체계(Supervisor)'가 필요한지, 아니면 '스타트업식 자율 협업(Swarm)'이 필요한지 먼저 고민해 보세요.
