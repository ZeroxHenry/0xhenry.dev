AI 에이전트가 잘못된 정보를 말했습니다. 확인해 보니 검색된 '문서' 자체가 틀렸습니다. 
자, 이제 이 문서를 **어디서, 누가, 언제** 우리 시스템에 넣었는지 바로 찾아낼 수 있나요?

대부분의 RAG 시스템은 `Source` 필드에 파일명 정도만 기록합니다. 하지만 프로덕션 환경에서는 그것만으로 부족합니다. 답변의 모든 구성 요소(데이터, 모델, 프롬프트)의 족보를 캐는 **데이터 계보(Data Lineage)** 추적이 필수입니다.

---

### 왜 계보 추적이 어려운가?

AI 답변은 '합성물'이기 때문입니다.
- **Input**: 사용자 질문
- **Retrieval**: 5개의 서로 다른 문서 조각(Chunks)
- **Processing**: 프롬프트 템플릿 v1.2
- **Model**: GPT-4o-2024-05-13

이 중 하나만 오염되어도 결과는 망가집니다. 특히 데이터 조각(Chunk)이 수백만 개가 넘어가면, 특정 답변에 영향을 준 조각이 원본 파일의 몇 페이지에서 나왔는지 역추적하는 것은 불가능에 가깝습니다.

---

### 계보 추적 시스템의 3대 핵심 요소

#### 1. 유니크 청크 ID (Unique Chunk Identifier)
벡터 DB에 저장되는 모든 조각에 원본 문서의 해시, 위치(Page/Line), 가공 시점(Timestamp)을 포함한 고유 ID를 부여해야 합니다.

#### 2. 프롬프트 버전 지표 (Prompt Versioning)
답변이 생성될 때 사용된 프롬프트의 Git Commit Hash나 버전 번호를 메타데이터로 남겨야 합니다. "이 답변은 3월 2일자 프롬프트로 만들어졌군"이라고 알 수 있도록요.

#### 3. 트래픽 태깅 (Response Metadata)
사용자에게 나가는 마지막 JSON 응답에 `reference_ids`를 포함시키세요.

```json
{
  "answer": "...",
  "debug": {
    "model_version": "gpt-4o-0513",
    "prompt_id": "p_129",
    "retrieved_chunks": ["doc_77_p12", "doc_89_p5"]
  }
}
```

---

### 사고 발생 시 역추적 워크플로우

1. **답변 신고**: 유저가 답변이 틀렸다고 리포트함.
2. **Metadata 조회**: 해당 답변의 `retrieved_chunks` ID를 확인.
3. **Lineage DB 검색**: `doc_77_p12` 조각의 원본이 `내부_규정_v2.pdf`이고 2024년 1월에 업데이트된 것을 발견.
4. **조치**: 원본 데이터를 수정하거나, 해당 조각을 벡터 DB에서 삭제(Purge).

---

### Henry의 조언: "추적할 수 없는 것은 고칠 수 없다"

데이터 계보는 '보험'과 같습니다. 평소에는 비용처럼 느껴지지만, AI가 중요한 비즈니스 결정을 틀렸을 때 원인을 5분 안에 파악하고 수정할 수 있게 해주는 유일한 장치입니다. 성능(Accuracy)을 높이는 것만큼이나 **'실패의 원인을 밝히는 속도'**가 중요하다는 것을 기억하세요.

---

### 결론

AI 투명성은 단순히 모델의 가중치를 공개하는 것이 아닙니다. **데이터가 어떻게 답변으로 흘러갔는지** 그 길을 투명하게 유지하는 것이 진짜 투명성입니다. 지금 여러분의 RAG 답변 뒤에 숨겨진 '족보'를 확인해 보세요.
