### 서론

여러 에이전트가 함께 작업할 때, 수평적인 구조는 때로 혼란을 야기합니다. 에이전트들이 서로 같은 말을 반복하거나, 업무를 넘기지 못하고 제자리걸음을 하는 '전달 루프'에 빠지기도 하죠. 다중 에이전트 시스템을 안정적으로 확장하려면 반드시 관리 계층이 필요합니다.

**관리자 패턴(Supervisor Pattern)**은 오직 다른 에이전트들을 관리하고 지시하는 일만 담당하는 중앙의 고수준 에이전트를 도입하는 방식입니다.

---

### 관리자 패턴의 작동 원리

관리자를 프로젝트 매니저(PM)라고 생각해 보세요:

1.  **목표 수신**: 사용자가 복잡한 결과물을 요청합니다. (예: "테스트 코드가 포함된 리액트 로그인 페이지를 만들어줘.")
2.  **할당 (Delegation)**: 관리자는 과업을 분석하고 첫 번째 전문가를 선택합니다. (예: "리서처 에이전트, 리액트 인증에 대한 최신 모범 사례를 찾아와.")
3.  **검토 (Review)**: 전문가가 작업을 마치면 다른 에이전트가 아닌 **관리자**에게 보고합니다.
4.  **다음 단계 결정**: 관리자는 결과물을 검토합니다. 내용이 좋으면 다음 전문가를 호출하고("코더 에이전트, 이제 코드를 작성해"), 내용이 부실하면 다시 해오라고 지시합니다.
5.  **종료 (Termination)**: 관리자가 모든 과정이 완료되었다고 판단하면 최종 답변을 사용자에게 제시합니다.

---

### 기업용 에이전트의 골드 표준인 이유

-   **특화된 전문가**: 하위 에이전트들은 프로젝트 전체를 알 필요 없이 아주 작고 구체적인 작업에만 집중할 수 있습니다. 이는 속도와 정확도를 모두 높입니다.
-   **품질 제어**: 관리라는 중앙 병목(Bottleneck)을 둠으로써, 부실한 데이터가 다음 단계로 넘어가는 것을 완벽히 차단할 수 있습니다.
-   **토큰 절약**: 하위 에이전트들은 전체 대화 기록을 공유할 필요 없이 관리자가 전달해 주는 특정 컨텍스트만 받으면 되므로 비용이 절감됩니다.

---

### LangGraph에서의 구현

LangGraph는 이러한 구조에 최적화되어 있습니다. 관리자 역할을 하는 '라우터(Router)' 노드를 정의하면 됩니다.

```python
from langgraph.graph import END

def supervisor_node(state):
    # 다음에 어떤 에이전트가 가야 할지 결정하는 로직
    if "research" not in state:
        return "researcher"
    if "code" not in state:
        return "coder"
    return END

# 그래프 엣지 구조 예시:

# workflow.add_conditional_edges("supervisor", ...)
```

### 요약

관리자 패턴은 **권한 위임과 통제**에 관한 것입니다. "다음에 무엇을 할까?"라는 책임을 개별 전문가 에이전트가 아닌 전담 관리자에게 부여함으로써, 훨씬 더 안정적이고 디버깅하기 쉬운 시스템을 구축할 수 있습니다.
