### 서론

지난 10년 동안 프론트엔드를 개발한다는 것은 사용자가 할 수 있는 모든 행동을 예측하고, 그에 맞는 컴포넌트를 하드 코딩(Hard-coding)하는 것을 의미했습니다. 날씨 위젯이 필요하다면 개발자가 미리 `<WeatherWidget />`을 만들어 두어야 했고, 주식 티커가 필요하다면 누군가 `<StockTicker />`를 코딩해야만 했죠.

하지만 2026년 현재, 우리는 **생성형 UI (Generative UI)** 시대에 진입했습니다. AI 모델은 더 이상 텍스트만 반환하지 않습니다. 이제는 실제 작동하는, 상호작용 가능한 코드 컴포넌트를 브라우저로 직접 스트리밍합니다.

---

### 생성형 UI란 무엇인가요?

생성형 UI는 사용자 인터페이스(UI)가 개발자에 의해 사전에 정의된 정적인 형태가 아니라, 현재의 맥락과 사용자의 '의도'에 따라 동적으로 렌더링되는 개념을 말합니다.

예를 들어 챗봇에게 이렇게 묻는다고 상상해 보세요. *"오늘 날씨는 어떻고, 그게 애플 주가에 어떤 영향을 미치고 있어?"*
-   **과거의 AI (텍스트 전용)**: 날씨와 주식 가격을 설명하는 긴 텍스트 문단을 반환합니다.
-   **생성형 UI**: 5일 치 예보 슬라이더가 있는 완벽한 '날씨 컴포넌트'와 실시간으로 스크롤할 수 있는 '주식 차트 컴포넌트'를 채팅창 안에 직접 렌더링해서 보여줍니다.

---

### 작동 원리: LLM을 통한 서버 주도 UI

이러한 마법은 Vercel의 AI SDK와 같은 프론트엔드 프레임워크를 통해 구현됩니다. 일반적인 작업 흐름은 다음과 같습니다:

1.  **함수 호출 (Function Calling)**: LLM이 특정 데이터(예: 항공편 정보)를 시각적으로 보여주어야 한다고 스스로 판단합니다. 그리고 `flight_tracker` 함수를 요청하는 구조화된 JSON 객체를 출력합니다.
2.  **서버 측 실행 (Server-Side Execution)**: 백엔드가 외부 API를 호출하여 실시간 항공 데이터를 가져옵니다.
3.  **컴포넌트 스트리밍 (Component Streaming)**: 서버는 데이터를 단순한 JSON 형태로 클라이언트에 보내는 대신, 해당 데이터가 주입된 React Server Component (RSC)를 동적으로 렌더링하여 HTML과 인터랙션 코드(Hydration Logic) 자체를 사용자의 화면으로 즉각 스트리밍합니다.

---

### '대시보드' 시대의 종말

생성형 UI는 전통적인 B2B 대시보드를 죽이고 있습니다.
특정 차트 하나를 찾기 위해 5단계의 복잡한 메뉴를 뒤져야 할 이유가 있을까요? 그냥 *"2분기 대비 3분기 매출을 지역별로 그룹화해서 보여줘"*라고 타이핑하면 화면에 고도로 맞춤화된 인터랙티브 차트가 뿅 하고 나타나는 데 말입니다.

인터페이스는 이제 복잡하고 딱딱한 '지도'가 아니라, 사용자의 의도에 맞춰 흐르는 '대화' 그 자체가 되었습니다.

---

### 요약

생성형 UI는 LLM의 무한한 지식과 현대 웹 앱의 구조화된 인터랙션 경험을 이어주는 다리입니다. AI가 단순한 문자열이 아니라 컴포넌트 공간 자체를 반환하도록 허용함으로써, 우리는 좁은 '채팅창'의 한계를 벗어나 웹 전체를 동적인 캔버스로 바꾸고 있습니다.
