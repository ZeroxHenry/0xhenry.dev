"코드 하나 안 건드렸는데, AI 답변이 갑자기 이상해요." 

LLM 기반 서비스를 운영하다 보면 겪게 되는 가장 황당하고 무서운 상황입니다. 원인은 보통 내가 아닌 외부에 있습니다. OpenAI, Anthropic 같은 제공사들이 **'몰래(?) 진행한 모델 업데이트'** 때문입니다.

이를 업계에서는 **모델 드리프트(Model Drift)** 또는 **모델 퇴보(Model Regression)**라고 부릅니다.

---

### 업데이트가 왜 독이 될까?

제공사들은 "성능이 향상되었습니다"라고 광고하지만, 실상은 다릅니다.
1. **정렬(Alignment) 강화**: 특정 단어 사용을 금지하거나 안전 가드레일을 높이면서, 기존에 잘 작동하던 유연한 시나리오가 막힙니다.
2. **비용 효율화**: 모델 내부의 파라미터를 압축하면서 미세한 추론 능력이 떨어질 수 있습니다.
3. **프롬프트 민감도 변화**: 기존 모델에 최적화된 프롬프트 워딩이 새 버전에서는 완전히 다른 가중치로 인식됩니다.

---

### 해결책 1: 모델 버전 핀(Version Pinning)

절대 `gpt-4o`나 `gpt-4-turbo` 같은 별칭(Alias)을 프로덕션에서 그대로 쓰지 마세요. 대신 **스냅샷 버전**을 명시해야 합니다.

- **나쁜 예**: `model="gpt-4o"` (언제든 바뀔 수 있음)
- **좋은 예**: `model="gpt-4o-2024-05-13"` (강제로 업데이트되지 않는 특정 시점 모델)

스냅샷을 쓰면 제공사가 새로운 모델을 내놓아도 내 서비스는 예전 모델로 평화롭게 유지됩니다.

---

### 해결책 2: 자동 응답 일관성 테스트 (Consistency Eval)

모델 업데이트가 감지되었을 때(혹은 정기적으로), 골든 데이터셋(Golden Dataset)을 이용해 '이전 답변'과 '현재 답변'의 **의미적 일관성**을 비교해야 합니다.

```python
# 업데이트 전/후 답변 비교 로직
def check_for_drift(old_answer, new_answer):
    similarity = semantic_distance(old_answer, new_answer)
    if similarity < 0.9:
        alert("🚨 모델 드리프트 감지! 답변 형식이 변했습니다.")
```

---

### 해결책 3: '모델 교체'를 위한 Shadow 테스트

5편에서 다룬 [Shadow Deployment](/ko/study/O_llmops/llm-shadow-testing)를 기억하시나요? 
제공사가 강제 업데이트를 공지했다면(예: "3개월 뒤에 이 스냅샷은 중단됩니다"), 즉시 새로운 스냅샷을 Shadow로 띄우고 기존 수치와 비교 분석을 시작해야 합니다.

---

### Henry의 팁: "제공사의 블로그보다 내 로그를 믿어라"

"이번 업데이트로 수학 능력이 20% 향상되었습니다"라는 공지사항보다, 우리 유저들의 복잡한 문장에서 발생하는 **'실제 통과율'**이 훨씬 중요합니다. 모델 업데이트 공지가 뜨면 축하하기보다 **'방어 모드'**를 먼저 가동하는 것이 시니어 LLM 엔지니어의 자세입니다.

LLM 인프라는 고정된 바닥이 아니라 **끊임없이 움직이는 파도** 위와 같습니다. **버전 핀링**과 **지속적 평가**라는 닻을 내리지 않으면, 여러분의 서비스는 어느 순간 산 대신 바다로 가고 있을지도 모릅니다.
